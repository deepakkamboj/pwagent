# Scheduler

The scheduler is powered by **`@bradygaster/squad-scheduler`**, a standalone package from the [deepakkamboj/squad](https://github.com/deepakkamboj/squad) fork (`packages/squad-scheduler`). Jobs are declared in a single `squad.schedule.json` file at your project root ŌĆö no per-job files, no separate daemon process.

## Quick start

**1. Create `squad.schedule.json` in your project root:**

```json
{
  "jobs": [
    {
      "id": "daily-triage",
      "name": "Daily Triage",
      "cron": "0 8 * * 1-5",
      "agent": "triage",
      "enabled": false
    }
  ]
}
```

**2. Start the scheduler:**

```powershell
pwagent scheduler start
```

The scheduler reads `squad.schedule.json`, registers all enabled jobs, and begins its 5-second tick loop. To use a different config file, set the `SQUAD_SCHEDULE_FILE` environment variable to the desired path.

## Job spec reference

| Field | Type | Required | Default | Description |
|---|---|---|---|---|
| `id` | string | yes | ŌĆö | Kebab-case identifier. Used for state tracking, locks, and log file names. |
| `name` | string | yes | ŌĆö | Human-readable label shown in `list` and `status` output. |
| `cron` | string | yes | ŌĆö | Standard 5-field cron expression (`minute hour dom month dow`). |
| `agent` | string | no | ŌĆö | Squad agent name. Runs `squad @<agent> <args>`. Mutually exclusive with `command`. |
| `args` | string | no | ŌĆö | Extra arguments appended after the agent name when using `agent`. |
| `command` | string | no | ŌĆö | Shell command to execute. Mutually exclusive with `agent`. |
| `enabled` | boolean | no | `true` | Set to `false` to skip the job without removing it. |
| `description` | string | no | ŌĆö | Shown in `pwagent scheduler list` output. |
| `maxRunSeconds` | number | no | `600` | Job is killed and marked `job_timeout` if it exceeds this duration. |
| `retryOnFailure` | boolean | no | `false` | Retry the job on non-zero exit. |
| `maxRetries` | number | no | `2` | Maximum retry attempts (when `retryOnFailure` is `true`). |
| `retryBackoffSeconds` | number | no | `30` | Seconds to wait between retry attempts. |
| `disableAfterFailures` | number | no | `5` | Consecutive failures before the job is automatically disabled. |
| `runOnStartup` | boolean | no | `false` | Fire the job once immediately when the scheduler starts, before the first tick. |

## Schedule expressions

Jobs use standard **5-field cron expressions**: `minute hour dom month dow`

| Field | Range | Special characters |
|---|---|---|
| minute | 0ŌĆō59 | `, - * /` |
| hour | 0ŌĆō23 | `, - * /` |
| day of month | 1ŌĆō31 | `, - * /` |
| month | 1ŌĆō12 | `, - * /` |
| day of week | 0ŌĆō7 (0 and 7 = Sunday) | `, - * /` |

**Examples:**

| Expression | Meaning |
|---|---|
| `0 8 * * 1-5` | 08:00 every weekday |
| `*/30 9-17 * * 1-5` | Every 30 minutes during business hours, weekdays |
| `0 * * * *` | Top of every hour |
| `0 2 * * *` | 02:00 every day |
| `0 17 * * 5` | 17:00 every Friday |
| `0 0 1 * *` | Midnight on the 1st of every month |

## Dispatch modes

Each job dispatches via exactly one of two modes: **agent** or **command**.

### Agent mode

Runs a Squad agent using `squad @<agent> <args>`. Use this for any job that should invoke an AI agent.

```json
{
  "id": "daily-triage",
  "name": "Daily Triage",
  "cron": "0 8 * * 1-5",
  "agent": "triage",
  "args": "--project MyProject --max-items 20"
}
```

Resolves to: `squad @triage --project MyProject --max-items 20`

### Command mode

Runs an arbitrary shell command. Use this for scripts, CLI tools, or non-agent automation.

```json
{
  "id": "hourly-flake-check",
  "name": "Hourly Flake Check",
  "cron": "0 * * * *",
  "command": "node scripts/check-flaky-tests.js --threshold 0.05"
}
```

`agent` and `command` are mutually exclusive ŌĆö specifying both is a validation error.

## CLI commands

| Command | Description |
|---|---|
| `pwagent scheduler start` | Start the scheduler in the foreground. Reads `squad.schedule.json` and begins the tick loop. |
| `pwagent scheduler stop` | Signal the running scheduler to shut down gracefully. |
| `pwagent scheduler list` | List all jobs with their id, name, cron expression, enabled state, and last-run status. |
| `pwagent scheduler status <id>` | Show detailed status for a single job: last run time, exit code, next scheduled run, retry count. |
| `pwagent scheduler logs <id>` | Stream the JSONL event log for a job. Defaults to the last 50 events. |

## Lifecycle events

Each job writes append-only JSONL to `~/.pwagent/scheduler/events/<id>.jsonl`. The scheduler itself writes to `~/.pwagent/scheduler/events/scheduler.jsonl`.

**Example job run (success):**

```jsonc
{ "ts": "2026-05-14T08:00:00Z", "type": "job_start",  "job": "daily-triage", "attempt": 1 }
{ "ts": "2026-05-14T08:01:43Z", "type": "job_end",    "job": "daily-triage", "exitCode": 0, "durationMs": 103200 }
```

**Example job run (failure with retry then auto-disable):**

```jsonc
{ "ts": "2026-05-14T08:00:00Z", "type": "job_start",        "job": "daily-triage", "attempt": 1 }
{ "ts": "2026-05-14T08:00:12Z", "type": "job_error",        "job": "daily-triage", "exitCode": 1, "stderr": "connection refused" }
{ "ts": "2026-05-14T08:00:42Z", "type": "job_retry",        "job": "daily-triage", "attempt": 2, "backoffSeconds": 30 }
{ "ts": "2026-05-14T08:01:10Z", "type": "job_error",        "job": "daily-triage", "exitCode": 1, "stderr": "connection refused" }
{ "ts": "2026-05-14T08:01:10Z", "type": "job_auto_disabled","job": "daily-triage", "reason": "consecutive failures: 5" }
```

**Event types:**

| Type | Emitted when |
|---|---|
| `scheduler_start` | Scheduler process starts. |
| `scheduler_stop` | Scheduler process shuts down. |
| `job_start` | A job begins execution. Includes `attempt` number. |
| `job_end` | A job completes successfully. Includes `exitCode` and `durationMs`. |
| `job_error` | A job exits with a non-zero code. Includes `exitCode` and `stderr`. |
| `job_timeout` | A job is killed after exceeding `maxRunSeconds`. |
| `job_retry` | A job is queued for retry. Includes `attempt` and `backoffSeconds`. |
| `job_auto_disabled` | A job is disabled after exceeding `disableAfterFailures`. |

## Tick loop guarantees

- **5-second tick** ŌĆö the scheduler wakes every 5 seconds and evaluates which jobs are due.
- **Per-job locks** ŌĆö a lock prevents the same job from running concurrently with itself across ticks.
- **Atomic state writes** ŌĆö `state.json` is written via a temp-file rename; no partial reads.
- **JSONL event stream** ŌĆö one append-only file per job at `~/.pwagent/scheduler/events/<id>.jsonl`.
- **Hot-reload** ŌĆö `fs.watch` monitors `squad.schedule.json` with a 300 ms debounce; changes are applied on the next tick without restarting the scheduler.
- **Retry with backoff** ŌĆö when `retryOnFailure` is true, failed jobs are retried up to `maxRetries` times, waiting `retryBackoffSeconds` between attempts.
- **Auto-disable** ŌĆö after `disableAfterFailures` consecutive failures, the job's `enabled` state is set to `false` in memory (the source `squad.schedule.json` is not modified).

## Platform service installer

To run the scheduler automatically on login and across reboots:

```powershell
pwagent service install
```

| Platform | Mechanism |
|---|---|
| Windows | Task Scheduler XML (`schtasks /Create`) |
| macOS | launchd plist (`launchctl load`) |
| Linux | systemd `--user` unit (`systemctl --user enable --now`) |

```powershell
pwagent service status      # check whether the service is installed and running
pwagent service uninstall   # remove the OS-level registration
```

The service runs `pwagent scheduler start` ŌĆö the same code path as running it in the foreground.

## Portal integration

The pwagent portal reads the same state files that the CLI writes:

- **`/scheduler`** ŌĆö shows overall scheduler status, uptime, last tick time, and a summary of all jobs.
- **`/jobs`** ŌĆö lists all jobs from `squad.schedule.json` with live status pulled from `~/.pwagent/scheduler/state.json`.
- **`/jobs/<id>`** ŌĆö tails `~/.pwagent/scheduler/events/<id>.jsonl` via SSE for a live event stream.

No separate API server is required ŌĆö the portal reads state files directly.
