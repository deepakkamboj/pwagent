# Architecture

pwagent is **Squad's design with our runtime**. The eleven Squad benefits port into the binary as first-class modules or filesystem conventions — same charter format, same routing.md / ceremonies.md / team.md files, but no upstream `squad.agent.md` coordinator manifest at run time. The coordinator logic is a TypeScript module inside the binary.

## Two daily-driver flows, one content base

The CLI now has **two distinct entry points**, each optimised for its context:

| Flow | Entry | What happens |
|---|---|---|
| **Chat** (interactive) | `pwagent` (no args, TTY) | Scaffolds `.pwagent/` (if missing), mirrors to `.squad/`, spawns `@bradygaster/squad-cli` which launches GitHub Copilot CLI with our 13 agents pre-loaded. Native Copilot CLI UX — banner, slash menu, autocomplete, streaming, multi-line input. |
| **CI / unattended** | `pwagent run <agent>` | Uses pwagent's own coordinator + `@github/copilot-sdk` directly. No Copilot CLI dependency on CI runners. Same charters, same skills, headless. |

Both flows share the embedded content (charters / skills / routing / team / ceremonies / master-prompt). The only difference is the surface through which the user (or scheduler) invokes them.

## Hard rules

- **Squad design, Copilot CLI as host (for chat).** Charters live in `.pwagent/agents/<name>/charter.md`; `.squad/` is an auto-generated mirror so Squad CLI can discover them. User edits `.pwagent/`; never touches `.squad/`.
- **Embedded by default.** All 13 charters and 60+ skill guides ship inside the binary. Works zero-config in any directory.
- **Two surfaces, one runtime contract.** `pwagent run` uses our coordinator. `pwagent` (chat) delegates to Squad → Copilot CLI. The agents themselves don't care which surface invoked them.
- **Scheduler is a sub-command**, not a separate daemon. Same logs, same config, same process.
- **Three independent layers.** CLI, portal, and docs each function alone.
