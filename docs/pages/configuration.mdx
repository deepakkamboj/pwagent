# Configuration

The user config lives at `~/.pwagent/config.json`. It's validated by [zod](https://zod.dev/) on every read; bad shapes refuse to load.

## repos — repository registry

The `repos` array replaces the old per-skill `repos.config.md` pattern. Every repo that agents may create bugs in, open PRs against, or run axe scans for should have an entry here.

| Field | Required | Description |
|---|---|---|
| `name` | ✓ | Repo name — matched against `git remote get-url origin` for auto-detection |
| `path` | ✓ | Local clone path |
| `type` | ✓ | `"ado"` or `"github"` |
| `defaultBranch` | ✓ | Branch PRs target (usually `"main"`) |
| `remoteUrl` | | Full ADO/GitHub remote URL — enables auto-detect when agents infer context from `git remote` |
| `ado.org` | | ADO org URL — overrides global `ado.org` for repos in a different org |
| `ado.project` | | ADO project — overrides global `ado.project` |
| `ado.areaPath` | | Area path for bug creation (e.g. `OneCRM\Client\UnifiedClient`) |
| `ado.iteration` | | Iteration path (e.g. `OneCRM\Current`) |
| `ado.bugTags` | | Default tags applied to every accessibility bug created in this repo |
| `ado.defaultAssignee` | | Default `assignedTo` value (alias or email) |
| `ado.serviceTreeId` | | ServiceTree GUID — passed to S360 for action item linking |
| `ado.mcpServer` | | MCP server ID to use (`"dynamicscrm-repo"` or `"msazure-repo"`) |
| `ado.productAreas` | | Monorepo sub-directory → area path mappings (see `power-platform-ux` example) |

**Auto-detection** — when an agent shells out `git remote get-url origin`, pwagent matches the returned URL against `repos[*].remoteUrl`. If a match is found, the agent uses that entry's `ado.*` fields without asking. If no match is found, the agent asks which repo to use — it never guesses.

## ado.defaults — accessibility bug defaults

Applied to every bug created by `pwagent-a11y` and `pwagent-s360`, unless a per-repo `ado.bugTags` / per-call override is present.

| Field | Description |
|---|---|
| `bugType` | Work item type — always `"Bug"` |
| `bugSeverity` | Default severity (`"2 - High"`) |
| `baseTags` | Tags added to every bug (`["accessibility", "a11y", "automated"]`) |
| `priorityBySlaState.OutOfSla` | Priority for S360 out-of-SLA items — `"1 - Critical"` |
| `priorityBySlaState.ApproachingSla` | `"2 - High"` |
| `priorityBySlaState.InSla` | `"3 - Medium"` |

## Editing

Three paths:

### CLI

```powershell
pwagent config view
pwagent config get providers.copilot.model
pwagent config set providers.copilot.model claude-haiku-4-5
pwagent config path                           # print the file path
```

### Portal

Open `http://127.0.0.1:7337/config`. The portal renders the config as a form with per-field zod validation. The **diff preview** shows exactly what changes before you save. Save writes atomically (`tmp` + rename).

### Hand-edit

```powershell
notepad "$env:USERPROFILE\.pwagent\config.json"
```

Bad JSON or schema violations cause the next `pwagent` command to exit with a structured error. The config is **not** silently reset — you fix it or revert.

## Atomic writes

Every config write follows the same pattern:

1. Read current config
2. Apply change (CLI sets one field; portal applies a full new object)
3. Validate with zod
4. Write to `~/.pwagent/config.json.tmp`
5. `fs.rename` over `~/.pwagent/config.json`

Step 5 is atomic on every supported OS — even if pwagent crashes mid-step, you either have the old file or the new file, never a half-written one.

## Sensitive fields

The CLI redacts sensitive fields when printing:

```
$ pwagent config view
{
  "notifications": {
    "report": {
      "webhookUrl": "<redacted>"
    }
  }
}
```

Add field names to `audit.redactKeys` to extend redaction:

```jsonc
{
  "audit": {
    "redactKeys": ["webhookUrl", "internalToken"]
  }
}
```

## Migration

When the schema bumps, pwagent reads the old config, applies the migration, writes the new one, and bumps `schema:`. Migrations live in `cli/src/config/migrations/` and run in order. **No manual migration needed.**
