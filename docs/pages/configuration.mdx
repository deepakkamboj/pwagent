# Configuration

The user config lives at `~/.pwagent/config.json`. It's validated by [zod](https://zod.dev/) on every read; bad shapes refuse to load.

## Full schema

```jsonc

  "schema": 1,

  "providers": {
    "copilot": {
      "model": "claude-sonnet-4.5",
      "idleTimeoutMs": 300000,
      "toolTimeoutMs": 120000,
      "toolOutputMaxKb": 200

  "perAgent": {
    "triage":   { "model": "claude-sonnet-4.5" },
```

## Editing

Three paths:

### CLI

```powershell
pwagent config view
pwagent config get providers.copilot.model
pwagent config set providers.copilot.model claude-haiku-4-5
pwagent config path                           # print the file path
```

### Portal

Open `http://127.0.0.1:7337/config`. The portal renders the config as a form with per-field zod validation. The **diff preview** shows exactly what changes before you save. Save writes atomically (`tmp` + rename).

### Hand-edit

```powershell
notepad "$env:USERPROFILE\.pwagent\config.json"
```

Bad JSON or schema violations cause the next `pwagent` command to exit with a structured error. The config is **not** silently reset — you fix it or revert.

## Atomic writes

Every config write follows the same pattern:

1. Read current config
2. Apply change (CLI sets one field; portal applies a full new object)
3. Validate with zod
4. Write to `~/.pwagent/config.json.tmp`
5. `fs.rename` over `~/.pwagent/config.json`

Step 5 is atomic on every supported OS — even if pwagent crashes mid-step, you either have the old file or the new file, never a half-written one.

## Sensitive fields

The CLI redacts sensitive fields when printing:

```
$ pwagent config view
{
  "notifications": {
    "report": {
      "webhookUrl": "<redacted>"
    }
  }
}
```

Add field names to `audit.redactKeys` to extend redaction:

```jsonc
{
  "audit": {
    "redactKeys": ["webhookUrl", "internalToken"]
  }
}
```

## Migration

When the schema bumps, pwagent reads the old config, applies the migration, writes the new one, and bumps `schema:`. Migrations live in `cli/src/config/migrations/` and run in order. **No manual migration needed.**
